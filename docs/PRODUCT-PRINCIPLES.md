# Product Principles

Ces principes guident toutes les décisions produit de VISITOR-OS.

## 1. Simplicité avant puissance

Une fonctionnalité simple, fiable et comprise vaut mieux qu'une fonctionnalité avancée mais fragile.

Le produit doit être utile sans formation.

## 2. Une action principale par écran

Chaque écran doit répondre immédiatement à une question :

```text
Quelle est l'information la plus importante ici ?
```

Si un écran demande trop d'effort cognitif, il doit être simplifié.

## 3. Le visiteur ne doit jamais subir le produit

Le widget doit rester :

- discret ;
- rapide ;
- rassurant ;
- non intrusif ;
- utile.

La collecte d'informations doit être progressive et justifiée.

## 4. L'administrateur doit comprendre vite

L'admin doit prioriser :

- les conversations importantes ;
- les prospects chauds ;
- les relances ;
- les erreurs ;
- les performances essentielles.

VISITOR-OS ne doit pas transformer l'administrateur en analyste de données.

## 5. La connaissance prime sur l'IA

L'IA ne doit pas inventer.

Ordre produit attendu :

```text
Configuration métier -> FAQ -> Documents -> Knowledge Base -> IA -> Escalade
```

Le produit doit privilégier une réponse vérifiée à une réponse spectaculaire.

## 6. Multi-activité, pas spécifique à un client

Le coeur produit ne doit jamais contenir une logique codée pour un seul métier.

Une activité doit être portée par :

- configuration ;
- documents ;
- règles ;
- modules optionnels.

## 7. Le coût doit rester prévisible

Les fonctionnalités qui augmentent fortement les coûts d'infrastructure, d'IA ou de maintenance doivent être optionnelles ou retardées.

## 8. Chaque module doit avoir une raison claire

Un module existe seulement s'il apporte une valeur autonome.

Un module ne doit pas être ajouté parce qu'il est techniquement intéressant.

## 9. Le produit doit rester vendable

Une fonctionnalité doit pouvoir être expliquée simplement à un client.

Si la proposition de valeur est confuse, la fonctionnalité n'est pas prête.

## 10. Ne pas confondre plateforme et complexité

VISITOR-OS peut être une plateforme sans devenir une architecture lourde.

La modularité doit réduire le risque, pas l'augmenter.
